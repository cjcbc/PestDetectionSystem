# 后端接口开发计划

## 概述

基于数据库表设计，规划实现文章评论、点赞和预警通知相关的后端接口。

**优先级说明：**
- 🔴 **P0（必做）** - 核心功能，立即实现
- 🟡 **P1（重要）** - 完整功能，尽快实现  
- 🟢 **P2（可选）** - 优化功能，后续迭代

## 一、文章相关接口

### 1.1 文章点赞 🔴

**接口定义**

```http
POST /api/article/{articleId}/like
Authorization: Bearer <token>
```

**功能**
- 点赞博文（或取消点赞，通过状态判断）
- 自动更新 article 表的 like_count

**请求参数**

```json
{}
```

**返回值**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "articleId": 123456789,
    "liked": true,
    "likeCount": 24
  }
}
```

**实现步骤**
1. 检查用户是否已点赞（查询 article_like 表）
2. 若已点赞则删除记录（取消点赞），否则插入新记录
3. 更新 article 表的 like_count 字段
4. 返回当前点赞状态和总数

**关键 SQL**
```sql
-- 检查是否已点赞
SELECT * FROM article_like WHERE article_id = ? AND user_id = ?

-- 插入点赞记录
INSERT INTO article_like (id, article_id, user_id, created_time) 
VALUES (?, ?, ?, ?)

-- 删除点赞记录
DELETE FROM article_like WHERE article_id = ? AND user_id = ?

-- 更新点赞数
UPDATE article SET like_count = (
  SELECT COUNT(*) FROM article_like WHERE article_id = ?
) WHERE id = ?
```

### 1.2 文章评论列表 🔴

**接口定义**

```http
GET /api/article/{articleId}/comments
  ?page=1&pageSize=20&sortBy=newest
```

**参数说明**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `articleId` | `long` | 是 | 文章ID |
| `page` | `int` | 否 | 页码，默认 1 |
| `pageSize` | `int` | 否 | 每页数量，默认 20，最大 100 |
| `sortBy` | `string` | 否 | 排序方式：newest（最新）\| hottest（最热） |

**返回值**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "total": 150,
    "pageNum": 1,
    "pageSize": 20,
    "pages": 8,
    "list": [
      {
        "id": 987654321,
        "userId": 123456,
        "username": "农民张三",
        "userImage": "data:image/jpeg;base64,...",
        "content": "这篇文章写得很好，对我很有帮助！",
        "likeCount": 5,
        "replyCount": 2,
        "isLiked": false,
        "createdTime": 1704067200000,
        "replies": [
          {
            "id": 987654322,
            "userId": 789012,
            "username": "博主李四",
            "userImage": "data:image/jpeg;base64,...",
            "content": "@农民张三 感谢你的反馈，欢迎常来交流！",
            "likeCount": 1,
            "isLiked": false,
            "createdTime": 1704070800000
          }
        ]
      }
    ]
  }
}
```

**实现步骤**
1. 根据 article_id 分页查询顶级评论（parent_id IS NULL）
2. 对每条评论查询最多 3 条最新回复（嵌套查询）
3. 从 comment_like 表判断当前用户是否已点赞各评论
4. 按排序参数返回结果

**关键 SQL**
```sql
-- 查询顶级评论（分页）
SELECT * FROM article_comment 
WHERE article_id = ? AND parent_id IS NULL AND status = 1
ORDER BY created_time DESC
LIMIT ?, ?

-- 查询某评论的回复（最多3条）
SELECT * FROM article_comment 
WHERE parent_id = ? AND status = 1
ORDER BY created_time ASC
LIMIT 3

-- 查询用户是否点赞过该评论
SELECT * FROM comment_like 
WHERE comment_id = ? AND user_id = ?
```

### 1.3 发布评论 🔴

**接口定义**

```http
POST /api/article/{articleId}/comments
Authorization: Bearer <token>
Content-Type: application/json
```

**请求参数**

```json
{
  "content": "这篇文章对我很有启发，感谢分享！",
  "parentId": null
}
```

**参数说明**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `content` | `string` | 是 | 评论内容，1-1000 字 |
| `parentId` | `long` | 否 | 父评论ID，NULL 表示顶级评论 |

**返回值**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 987654325,
    "articleId": 123456789,
    "userId": 123456,
    "username": "当前用户",
    "userImage": "data:image/jpeg;base64,...",
    "content": "这篇文章对我很有启发，感谢分享！",
    "parentId": null,
    "likeCount": 0,
    "replyCount": 0,
    "createdTime": 1704153600000
  }
}
```

**实现步骤**
1. 验证评论内容长度（1-1000 字）
2. 若 parentId 非空，验证父评论是否存在且未被删除
3. 插入新评论记录（status = 1 自动发布，或 status = 0 待审核）
4. 若有 parentId，则更新父评论的 reply_count
5. 返回刚插入的评论

**关键 SQL**
```sql
-- 插入评论
INSERT INTO article_comment 
  (id, article_id, user_id, parent_id, content, status, created_time, updated_time)
VALUES (?, ?, ?, ?, ?, 1, ?, ?)

-- 更新父评论的回复数
UPDATE article_comment SET reply_count = reply_count + 1 
WHERE id = ?
```

### 2.3 标记预警已读 🔴

**接口定义**

```http
POST /api/warning/{warningId}/read
Authorization: Bearer <token>
```

**功能**
- 标记用户对某预警已读
- 若已读过则不重复记录

**请求参数**

```json
{}
```

**返回值**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "warningId": 555666777,
    "userId": 123456,
    "readTime": 1704240600000
  }
}
```

**实现步骤**
1. 检查预警是否存在且已发布
2. 检查用户是否已读过（UNIQUE 约束防止重复）
3. 若未读则插入新记录，若已读则更新 readTime
4. 返回已读时间

**关键 SQL**
```sql
-- 检查是否已读
SELECT * FROM warning_read WHERE warning_id = ? AND user_id = ?

-- 插入已读记录，若冲突则跳过
INSERT IGNORE INTO warning_read (id, warning_id, user_id, read_time)
VALUES (?, ?, ?, ?)
```

### 2.4 获取未读预警数量 🔴

**接口定义**

```http
GET /api/warning/unread/count
Authorization: Bearer <token>
```

**功能**
- 获取当前用户未读预警的数量
- 前端消息中心可显示小红点或badge

**返回值**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "unreadCount": 3,
    "warnings": [
      {
        "id": 555666777,
        "title": "小麦赤霉病预警",
        "severity": 3,
        "publishTime": 1704240000000
      },
      {
        "id": 555666778,
        "title": "水稻螺纹螺旋体病预警",
        "severity": 2,
        "publishTime": 1704240100000
      },
      {
        "id": 555666779,
        "title": "玉米南方锈病预警",
        "severity": 2,
        "publishTime": 1704240200000
      }
    ]
  }
}
```

**实现步骤**
1. 查询用户已读的预警 ID 集合
2. 统计未读预警数量
3. 返回最近 3 条未读预警摘要，供消息中心显示

**关键 SQL**
```sql
-- 获取未读预警数和摘要
SELECT 
  COUNT(DISTINCT w.id) as unreadCount,
  GROUP_CONCAT(
    JSON_OBJECT('id', w.id, 'title', w.title, 'severity', w.severity, 'publishTime', w.publish_time)
    ORDER BY w.publish_time DESC
    SEPARATOR ','
  ) as warnings
FROM warning w
WHERE w.status = 1 
  AND w.id NOT IN (
    SELECT warning_id FROM warning_read WHERE user_id = ?
  )
LIMIT 3
```

## 四、技术实现建议

### 使用框架/工具
- **ORM**：MyBatis Plus（快速 CRUD）
- **分页**：PageHelper（简化分页逻辑）
- **雪花ID**：自定义生成器或第三方库（保证 ID 全局唯一）
- **缓存**：Redis（缓存热门文章、预警列表）

### 关键业务逻辑
1. **唯一性约束** - article_like、comment_like、warning_read 表中的 UNIQUE 索引
2. **计数器同步** - 点赞/取消点赞时同步更新对应 count 字段
3. **级联删除** - 删除评论时自动删除子评论和所有点赞记录
4. **异步更新** - viewCount 和 readTime 可使用 MQ 异步更新，提高接口响应速度

### 安全考虑
1. **权限验证** - 管理员接口需要 role = 0 的校验
2. **内容审核** - 对评论内容做关键词过滤和审核流程
3. **频率限制** - 对评论、点赞接口做单用户请求频率限制
4. **SQL 注入防护** - 使用参数化查询，不手写 SQL
